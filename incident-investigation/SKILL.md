# incident-investigation

用于线上故障排查的 Skill。

## 何时使用

当用户明确提到以下需求时使用：

- 分析线上问题
- 排查故障
- 定位异常原因
- 分析告警
- 看日志 / 指标后给初步判断

## 先读哪些文件

命中本 Skill 后，按需读取以下文件：

1. `references/triage-playbook.md`
   - 先看这里，理解排障原则、顺序和边界。
2. `references/metrics-checklist.md`
   - 当需要判断该看哪些指标时读取。
3. `references/tools.md`
   - 查看有哪些可用的脚本工具可以调用。
4. `references/output-template.md`
   - 输出最终结果前读取，确保结构稳定。

## 任务目标

帮助 Agent 在排障场景中：

1. 先确认故障对象、环境、时间窗口和现象
2. 收集日志、指标、最近变更等证据
3. 基于证据归纳可能原因
4. 输出结构化分析结果
5. 对高风险操作保持克制，必要时先请求确认

## 执行边界

- 负责收集证据、归纳问题、输出建议
- 不直接执行重启、回滚、删数据、改线上配置等高风险动作
- 如果需要外发通知或执行修复动作，必须先请求确认
- 如果证据不足，优先补问，不得强行归因

## 执行流程

1. 确认服务名、环境、时间窗口、故障现象
2. 读取 `references/triage-playbook.md`
3. 如需判断指标范围，读取 `references/metrics-checklist.md`
4. 优先收集日志、指标、最近变更这三类证据
5. 至少结合两类证据再做初步判断
6. 输出前读取 `references/output-template.md`
7. 按模板输出：故障现象、已知事实、初步判断、建议验证动作、风险提示

## 最小输入要求

至少具备以下信息中的两项：

- 服务名
- 环境
- 时间窗口
- 故障现象
- 告警标题

如果不满足最小输入要求，先补问关键上下文。

## 注意事项

- 没有证据不要下结论
- 不编造日志、指标和链路信息
- 推断必须说明依据
- 低置信度结论必须明确标注不确定性
- 不要默认“重启一下试试”
